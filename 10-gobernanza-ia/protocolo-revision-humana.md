# Protocolo de Revisión Humana para IA

Este documento define cuándo, cómo y por quién deben revisarse las salidas generadas por inteligencia artificial, agentes o automatizaciones dentro de ARNI Consulting.

## Objetivo

Asegurar que todo contenido, análisis, propuesta, mensaje o modificación generada con apoyo de IA sea revisado antes de usarse en decisiones críticas, comunicación externa o entregables de cliente.

La revisión humana protege la calidad, la reputación, la confidencialidad y el control profesional de ARNI Consulting.

## 1. Principio general

Toda salida de IA debe considerarse preliminar hasta que una persona responsable la revise.

La IA puede apoyar en velocidad, estructura y análisis, pero la responsabilidad final corresponde a ARNI Consulting.

## 2. Cuándo se requiere revisión humana

La revisión humana es obligatoria antes de:

- Enviar correos a clientes, prospectos o aliados.
- Enviar propuestas comerciales.
- Compartir reportes con clientes.
- Publicar contenido institucional.
- Modificar datos críticos.
- Crear oportunidades comerciales definitivas.
- Cambiar prioridades o estados comerciales.
- Usar análisis para una decisión estratégica.
- Presentar recomendaciones finales.
- Compartir información con terceros.

## 3. Niveles de revisión

### Nivel 1: Revisión ligera

Aplica para contenido interno de bajo riesgo.

Ejemplos:

- Resúmenes internos.
- Listas de pendientes.
- Ideas preliminares.
- Borradores exploratorios.

Revisión requerida:

- Claridad.
- Coherencia.
- Utilidad.
- Ausencia de errores evidentes.

### Nivel 2: Revisión operativa

Aplica para documentos que pueden afectar seguimiento, CRM, Airtable o procesos comerciales.

Ejemplos:

- Minutas.
- Clasificación de oportunidades.
- Matrices A/B/C.
- Seguimientos B2B.
- Scoring de importadores.
- Roadmaps internos.

Revisión requerida:

- Exactitud de datos.
- Cliente correcto.
- Proyecto correcto.
- Próximo paso.
- Responsable.
- Riesgos.
- Supuestos.

### Nivel 3: Revisión directiva

Aplica para documentos externos, propuestas o decisiones sensibles.

Ejemplos:

- Propuestas comerciales.
- Reportes a cliente.
- Mensajes institucionales.
- Diagnósticos estratégicos.
- Cambios de alcance.
- Honorarios.
- Exclusiones.
- Decisiones de avance o descarte.

Revisión requerida:

- Alcance.
- Entregables.
- Riesgos.
- Exclusiones.
- Lenguaje de compromiso.
- Implicaciones comerciales.
- Confidencialidad.
- Alineación con ARNI.

## 4. Criterios de revisión

Toda revisión debe verificar:

| Criterio | Pregunta clave |
|---|---|
| Exactitud | ¿La información es correcta? |
| Fuente | ¿La información tiene respaldo o debe verificarse? |
| Cliente correcto | ¿Corresponde al cliente o proyecto adecuado? |
| Supuestos | ¿Están claramente identificados? |
| Riesgos | ¿Se señalan riesgos relevantes? |
| Alcance | ¿No promete más de lo debido? |
| Confidencialidad | ¿No expone información sensible? |
| Próximo paso | ¿La acción recomendada es clara? |
| Tono | ¿El lenguaje es profesional y adecuado? |
| Decisión | ¿La salida requiere aprobación antes de usarse? |

## 5. Roles sugeridos

| Rol | Responsabilidad |
|---|---|
| Usuario operativo | Revisa claridad, datos básicos y formato |
| Responsable comercial | Revisa oportunidad, prioridad y seguimiento |
| Responsable de proyecto | Revisa alcance, entregables y riesgos |
| Dirección ARNI | Aprueba propuestas, decisiones sensibles y comunicación estratégica |

## 6. Checklist antes de enviar contenido externo

Antes de enviar cualquier contenido generado con apoyo de IA a un cliente, prospecto o aliado, revisar:

- ¿El nombre del cliente es correcto?
- ¿El contexto es correcto?
- ¿No hay información inventada?
- ¿No hay promesas no autorizadas?
- ¿No hay datos confidenciales indebidos?
- ¿El tono es profesional?
- ¿El alcance está limitado?
- ¿Los próximos pasos son claros?
- ¿La persona responsable aprobó el envío?

## 7. Checklist para propuestas comerciales

Antes de enviar una propuesta generada o apoyada por IA, revisar:

- Contexto del cliente.
- Necesidad identificada.
- Objetivo del servicio.
- Alcance.
- Entregables.
- Exclusiones.
- Cronograma.
- Honorarios.
- Responsabilidades del cliente.
- Riesgos.
- Vigencia de la propuesta.
- Lenguaje de compromiso.
- Próximos pasos.

## 8. Checklist para datos y registros

Antes de modificar información en CRM, Airtable, GitHub u otra base, revisar:

- Registro correcto.
- Cliente correcto.
- Campo correcto.
- Valor actual.
- Nuevo valor propuesto.
- Justificación del cambio.
- Responsable que autoriza.
- Riesgo de duplicidad.
- Registro del cambio.

## 9. Criterios de rechazo

Una salida de IA debe rechazarse o corregirse si:

- Contiene información inventada.
- Mezcla clientes o proyectos.
- Promete resultados no controlables.
- Omite riesgos relevantes.
- Omite exclusiones.
- Presenta supuestos como hechos.
- Tiene tono inadecuado.
- Usa información confidencial indebidamente.
- No tiene próximo paso claro.
- Puede generar riesgo comercial o reputacional.

## 10. Registro de revisión

Cuando la revisión sea relevante, debe registrarse:

- Fecha.
- Documento o salida revisada.
- Responsable de revisión.
- Cambios realizados.
- Estado: aprobado, corregido o rechazado.
- Observaciones.

La revisión humana no debe verse como obstáculo. Es el mecanismo que permite usar IA con velocidad sin perder calidad, criterio ni responsabilidad profesional.
